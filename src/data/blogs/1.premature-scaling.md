Last year, while trying to improve the latency of an API service, I reached for Redis and EC2 auto scaling too early.

It felt like the mature choice. I had seen the pattern in tutorials, open-source projects, and production architecture diagrams: put a cache in front of slow work, add more compute when traffic rises, and call the system scalable.

That was not completely wrong. Redis and auto scaling are useful tools. The mistake was using them before I had proved I needed their failure modes.

<!--truncate-->

The real issue was simpler: the service repeated the same expensive calculation many times inside the same process. I did not need a shared cache yet. I needed to stop doing redundant work.

In Python, that can be as small as this:

```python
from functools import lru_cache

@lru_cache(maxsize=128)
def compute_price_rules(account_id: str) -> dict:
    return load_and_parse_rules(account_id)
```

That kind of local cache would have solved most of my latency problem without a new server, network hop, credential, eviction policy, deployment path, or on-call concern.

The point is not that `@lru_cache` is better than Redis. The point is that each scaling tool buys something specific and charges you for it.

Local caching is cheap and fast, but it is process-local. It does not share state across machines, and it disappears when the process restarts.

Redis gives you shared state, explicit TTLs, and cross-process coordination. It also gives you a new dependency, new operational failure modes, network latency, access control, backup questions, memory pressure, and cache invalidation risk.

Auto scaling gives you more capacity. It also makes deploys, cold starts, state placement, observability, and cost control more important.

None of those costs are a reason to avoid mature infrastructure. They are a reason to be precise about what problem you are solving.

Before adding infrastructure, I now ask a smaller set of questions:

- Is the slow work repeated inside one process?
- Is the data safe to cache?
- How stale can the answer be?
- What invalidates it?
- Does the cache need to be shared across instances?
- What happens when the cache is empty, wrong, full, or unavailable?
- Will this change make the system easier to operate, or only more impressive to describe?

Those questions matter in startups because time is scarce. They matter in enterprise systems because every new component becomes part of the control plane, the security surface, and the incident story.

Premature scaling is dangerous because it feels responsible. You are using serious tools. You are thinking about growth. You are doing work that looks like production engineering.

But production engineering is not the same as adding production-shaped components. It is choosing the smallest mechanism that preserves the right invariant under the expected pressure.

In this case, the invariant was simple: repeated requests should not recompute stable data unnecessarily.

The smallest useful mechanism was a bounded in-process cache.

Redis would have been appropriate later, if I needed shared cache state, coordinated invalidation, or consistent behavior across many workers. Auto scaling would have been appropriate later, if one process could no longer handle real measured load.

The lesson I took from it is not "avoid scaling." It is this:

Start by removing waste. Add infrastructure when the remaining problem genuinely needs infrastructure.

That is usually faster. It is usually safer. And it leaves fewer things for future operators to debug at 3 a.m.
